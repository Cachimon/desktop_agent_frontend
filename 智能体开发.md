> 本项目是一个用于**文件批量整理、跨应用搬运、模糊搜索**的 AI 桌面助手，以 **Electron-vite + Vue3 + FastAPI** 为核心技术栈，深度融合 **MCP 协议、Skills 技能系统与多智能体协作架构**，采用 **MySQL 作为长期存储引擎**，致力于打造一个本地优先、可扩展的 AI 智能体工作台。

## 一、项目概述

### 1.1 项目背景

随着 AI 大模型能力的持续演进，用户不再满足于“对话式问答”，而是期望 AI 能够真正操作电脑、执行多步骤任务。当前市面上虽有 ChatGPT Desktop、Copilot 等产品，但普遍存在：

- 仅提供聊天框，缺乏对桌面系统的深度控制能力；
    
- 文件操作、跨应用集成能力有限；
    
- 数据隐私存疑，多在云端完成推理；
    
- 缺乏模块化的技能扩展体系，难以适配个性化工作流。
    

本项目旨在构建一个**本地优先、可扩展、具备深度桌面操作能力**的 AI 智能体桌面助手，优先落地文件批量整理、跨应用搬运和模糊搜索三大高频痛点场景。

### 1.2 产品愿景

> **让 AI 成为一个真正能长期协作的桌面工作伙伴，而不仅仅是一次的对话框。**

核心设计理念：

- **本地优先**：对话、技能配置、工具调用记录等所有数据默认保存在本地；
    
- **可扩展性优先**：支持 MCP 协议接入外部工具、支持自定义 Skills；
    
- **安全可控**：高风险操作（Shell、文件删除、系统配置变更）须经用户确认。
    

### 1.3 目标用户

- **办公人员**：日常进行文件整理、邮件发送、数据跨应用搬运；
    
- **开发者**：需要自动化脚本、代码生成、日志分析；
    
- **普通电脑用户**：希望用自然语言完成文件搜索、系统操作。
    

### 1.4 核心场景定义

本产品第一版本聚焦三大场景：

1. **文件批量整理**：基于规则的自动归类、批量重命名、垃圾清理、压缩解压、重复文件清理；
    
2. **跨应用搬运**：浏览器 → Excel、邮件附件 → 指定文件夹、截图 → 聊天软件等跨应用数据流转；
    
3. **模糊搜索**：基于文件名、内容、修改时间等多维度的智能化文件定位。

## 三、核心模块设计

### 3.1 AI Agent 运行时 (LangGraph)

Agent 是智能体的决策中枢，负责接收用户意图、规划任务、调度 Skills 和 MCP 工具，最终完成执行。

```text
┌────────────────────────────────────────────────────────────┐
│                    LangGraph 工作流                          │
│  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────────┐  │
│  │ 用户输入 │ -> │ 意图识别 │ -> │ 任务规划 │ -> │ 工具调度   │  │
│  └────────┘    └────────┘    └────────┘    └────────────┘  │
│                                                  │          │
│                              ┌───────────────────┼──────────┤
│                              ▼                   ▼          │
│                        ┌──────────┐      ┌────────────┐    │
│                        │ 执行结果  │ <- │ 工具调用等待  │    │
│                        └──────────┘      └────────────┘    │
└────────────────────────────────────────────────────────────┘
```

核心特性（基于 LangGraph 1.1+）：

- **图节点编排**：使用 `StateGraph` 定义节点（PlanNode、ExecuteNode、SummarizeNode）和条件边；
    
- **类型安全的流式输出**：LangGraph v1.1 支持统一的 `StreamPart` 输出格式，每个 chunk 包含 `type`、`ns` 和 `data` 键，前端可精准识别消息类型；
    
- **记忆管理**：使用 `AsyncMySQLSaver`（LangGraph 官方提供的异步 MySQL 检查点保存器）持久化多轮对话上下文。每个会话的状态、工具调用记录、中间步骤均存储在 MySQL 的 `checkpoints` 表中，支持断点恢复和跨会话历史回溯；
    
- **Humans-in-the-Loop（HITL）** ：关键操作（删除文件、系统设置变更）执行前中断并请求用户确认。LangGraph 原生支持在任意节点检查和修改 Agent 状态。
    

**版本演进说明**：LangGraph 于 2025 年 10 月发布 v1.0，2026 年 1 月发布 1.0.7 进入生产就绪阶段，2026 年 3 月 19 日发布 1.1.3 稳定版。配套推出的 LangGraph CLI 0.4.12 工具链，实现了从项目初始化到 Docker 部署的全流程自动化支持。

### 3.2 Skills 技能系统（2026 年重新定义）

Skills 是由 Anthropic 于 2025 年确立的开放式 AI 代理构建标准，其本质是将复杂的 Prompt 工程、外部知识库（Reference）与执行逻辑（Script）封装为标准化的本地文件结构，核心是一个名为 `SKILL.md` 的文件。

> **2026 年重要趋势**：Agent Skills 已成为 2026 年初的核心话题，被 Cursor、Trae、Codex 等工具陆续支持。与 MCP 需要运行代码服务不同，Skills 是纯文件存储，轻量且可移植。微软 Azure AI Studio 已原生支持 Skills 格式，GitHub Copilot Workspace 中也已实验性支持。

本项目采用 **“渐进式披露”** （Progressive Disclosure）三层架构——该架构是 Agent Skill 的行业标准设计模式，区别于传统的“一次性加载所有内容”的方式，其核心价值在于将内容分层组织、按需加载：

|层级|名称|说明|加载时机|
|---|---|---|---|
|Layer 1|元数据层|技能名称、简短描述（~100 tokens）|启动时加载，识别可用技能|
|Layer 2|指令与 SOP 层|完整的工作流指令、操作规范|技能被触发时按需加载|
|Layer 3|资源与脚本层|代码文件、配置文件、参考文档|执行时动态调用|

**技能定义规范（SKILL.md）** ：每个技能由一个结构化的文件夹构成，核心文件是 `SKILL.md`。该格式支持跨客户端移植——同一套 Skill 定义可被不同模型或 Agent 实例加载，无需重复开发。

```yaml
# .agent/skills/file_organizer/SKILL.md
---
name: file_organizer
description: 按文件类型、修改时间自动整理文件夹
version: "1.0.0"
license: MIT
author: AI Assistant Team
---

# 指令
当用户请求整理文件夹时，按以下步骤执行：
1. 确认目标文件夹路径

3. 按文件扩展名分类

- target_structure: optional|string - 自定义目录结构

# 执行脚本
```python

    # 实现文件整理逻辑

### 3.4 多智能体 (Multi-Agent) 协作

为解决“全能型 Agent 的崩塌问题”（即单一上下文窗口承载过多工具定义导致推理效率下降），本项目采用 **主 Agent + SubAgent** 的协作架构，每个 SubAgent 聚焦单一领域能力，拥有最小权限集。

```


┌─────────────────────────────────────────────────────────────┐  
│ 主 Agent (规划决策) │  
│ 负责意图识别、任务拆解、结果聚合 │  
└───────────────────────────┬─────────────────────────────────┘  
│ 任务分发 (A2A 协议)  
┌───────────────────┼───────────────────┐  
▼ ▼ ▼  
┌───────────────┐ ┌───────────────┐ ┌───────────────┐  
│ 文件整理 Agent │ │ 跨应用 Agent │ │ 搜索 Agent │  
│ (File Skills) │ │(MCP Transport)│ │ (RAG + 检索) │  
└───────────────┘ └───────────────┘ └───────────────┘

```text

各 SubAgent 的专业化分工：

- **文件整理 Agent**：挂载 `file_organizer`、`batch_rename`、`duplicate_cleaner` 等技能，专注本地文件管理；
- **跨应用 Agent**：通过 MCP Server 调用外部应用（浏览器、Excel、邮件客户端），专注数据搬运场景；
- **搜索 Agent**：挂载向量检索和 RAG 能力，专注语义化模糊搜索。

SubAgent 之间通过 **A2A（Agent-to-Agent）协议**通信，该协议的质量已成为 2026 年 AI Agent 框架选型的重要衡量指标。

### 3.5 前端 (Vue3) 模块设计

基于 Electron-vite 模板的最佳实践，前端采用以下架构：

**主要界面及功能**：

| 模块 | 功能 | 技术要点 |
|------|------|---------|
| 对话主面板 | 流式 AI 对话 | SSE / WebSocket 接收 Agent 思考过程与最终回复 |
| 技能管理中心 | 技能包的添加/删除/启停 | 支持热加载符合 SKILL.md 规范的技能文件夹 |
| 工具调用可视化 | 展示 Agent 的工具调用链 | 实时展示 LLM 的 Thought → Action → Observation 流程 |
| 文件搜索结果视图 | 以列表/网格形式展示搜索结果 | 支持文件预览、批量选择操作 |
| 设置面板 | 模型配置、主题切换、数据路径 | 支持多 Provider（OpenAI 兼容格式） |

**关键组件实现**：

1. **流式聊天组件**：使用 `EventSource` 或 `fetch + ReadableStream` 接收 SSE 数据流，支持 LangGraph v1.1 的类型化 StreamPart 格式，自动识别增量文本、工具调用状态和错误消息；
2. **IPC 桥接模块**：通过 `contextBridge.exposeInMainWorld` 安全暴露 Electron Main 进程的系统调用 API（文件读写、Shell 执行、窗口控制）；
3. **本地持久化**：通过 `pinia-plugin-persistedstate` 实现对话历史、用户偏好的本地存储。

### 5.3 数据存储方案（MySQL 版）

|数据类型|存储方案|存储位置 / 表|
|---|---|---|
|对话历史|MySQL（InnoDB）|`conversations` 表（会话元数据 + 消息内容）|
|技能配置|YAML 文件 + MySQL 元数据缓存|磁盘 `~/.ai-assistant/.agent/skills/` + `skills_registry` 表|
|向量索引|LanceDB（磁盘优先）|`~/.ai-assistant/vector_store/`|
|用户设置|MySQL|`user_settings` 表|
|工具调用审计日志|MySQL|`audit_logs` 表（分区按日期）|
|Agent 检查点（LangGraph）|MySQL|`checkpoints` 表（由 LangGraph Saver 自动管理）|
|任务队列状态|Redis|内存 + 可选持久化 RDB/AOF|

**MySQL 表结构示例（核心表）**：

```sql
-- 对话会话表
CREATE TABLE conversations (
    id VARCHAR(36) PRIMARY KEY DEFAULT (UUID()),
    user_id VARCHAR(64) NOT NULL,
    title VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_created (user_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 消息表
CREATE TABLE messages (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    conversation_id VARCHAR(36) NOT NULL,
    role ENUM('user', 'assistant', 'system') NOT NULL,
    content LONGTEXT NOT NULL,
    tool_calls JSON,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
    INDEX idx_conversation (conversation_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 审计日志表（按月分区）
CREATE TABLE audit_logs (
    id BIGINT AUTO_INCREMENT,
    user_id VARCHAR(64),
    operation VARCHAR(64) NOT NULL,
    details JSON,
    ip_address VARCHAR(45),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
PARTITION BY RANGE (UNIX_TIMESTAMP(created_at)) (
    PARTITION p202601 VALUES LESS THAN (UNIX_TIMESTAMP('2026-02-01')),
    PARTITION p202602 VALUES LESS THAN (UNIX_TIMESTAMP('2026-03-01')),
    -- 动态添加
);
```

### 5.4 安全与沙箱

- **工具执行沙箱**：Shell 工具、代码解释器在独立进程中运行，实现路径隔离、权限隔离、资源隔离；
    
- **审计日志**：所有工具调用和敏感操作均记录审计日志；
    
- **模型数据隔离**：所有数据和 API Key 默认保存在本地，不上传任何云端服务；
    
- **数据库连接安全**：支持 TLS 加密连接，密码通过系统密钥链或环境变量注入，不硬编码；
    
- **备份策略**：提供简单的 `mysqldump` 脚本或集成自动备份功能（可选）。

## 七、技术展望与可扩展性

### 7.1 边缘端推理集成（未来计划）

2026 年小型语言模型在边缘端的部署已成为现实。量化后的 Phi-3 Mini（3.8B）、Gemma 3B 和 Llama 3.2 等模型在消费级硬件上即可高效运行。本项目可作为后续迭代方向：

- 通过 `llama.cpp` 或 ONNX Runtime 在本地加载小参数模型，处理敏感数据场景下的任务规划；
    
- 云端大模型负责复杂推理，边缘端小模型处理简单任务和本地文件理解，形成混合推理架构。
    

### 7.2 Skills 生态扩展

Skills 格式本质上是“可移植的 Agent 能力”，技能可以打包为压缩包分享，即插即用。本项目后续可建立技能市场，允许用户从社区安装第三方技能包，实现能力的按需扩展。

### 7.3 MCP Apps 界面能力

MCP Apps 的发布使 MCP Server 能够提供动态用户界面。未来可让 Agent 通过 MCP 调用时弹出特定界面（如文件选择器、表单填写界面），大幅提升交互体验。

### 7.4 数据库水平扩展

随着数据量增长，可平滑迁移至 TiDB 或其他 MySQL 兼容的分布式数据库，实现水平扩展，而无需修改业务代码。
